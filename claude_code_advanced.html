Topic 01 CLAUDE.md
CLAUDE.md とは何か：プロジェクトのルートに置く設定ファイル。Claude Code がセッション開始時に必ず読み込み、「このプロジェクトで何をするか・どう振る舞うか」を覚える。一度書けばセッションをまたいで指示が持続する。
何を書くか：プロジェクト概要・使用ファイル・出力ルール・禁止事項・よく使うコマンドのエイリアス。「毎回同じことを最初に説明している」なら CLAUDE.md に書くべき内容。
Mission 1-A ／ CLAUDE.md
D2C分析プロジェクト用の CLAUDE.md を設計・作成する
シナリオ
前ワークブックで作った sales_monthly.csv、customer_reviews.csv、competitor_info.txt を毎週使って分析している。毎回 Claude Code に「このフォルダには〜」と説明するのを自動化したい。プロジェクトルートに CLAUDE.md を置くと、次回から説明ゼロで作業が始まる。
ファイル構造（作業前）
d2c-analytics/
├── sales_monthly.csv
├── customer_reviews.csv
├── competitor_info.txt
└── product_spec.txt  ← CLAUDE.md をここに作る
タスク ／ 成果物
  • プロジェクト概要・ファイル一覧・出力ルールを含む CLAUDE.md を作成するCLAUDE.md
  • 「レポートは必ず outputs/ フォルダに保存」「グラフは PNG+日本語フォント対応」のルールを書く
  • 作成後に新しいセッションで claude を起動し、何も説明せず「売上サマリーを出して」と指示して動作確認する
着目ポイント：CLAUDE.md がないセッションとある セッションで最初の応答がどう変わるか比べてみること。
模範 CLAUDE.md
CLAUDE.md# D2C Analytics Project — ビタグロウ ## プロジェクト概要 D2Cコスメブランド「ビタグロウ セラム エクストリーム」の マーケティング分析プロジェクト。 ## データファイル - sales_monthly.csv : チャネル別月次売上（1〜6月） - customer_reviews.csv : 商品レビュー 20件（評価・本文・属性） - competitor_info.txt : 競合4社の調査メモ - product_spec.txt : 自社商品仕様書 ## 出力ルール（必ず守ること） - レポートファイルは outputs/ フォルダに保存する - グラフは PNG 形式・日本語フォント対応（文字化け禁止） - 数値は万円単位、小数点1桁 - CSV 出力時のエンコードは UTF-8-BOM（Excel対応） ## よく使う操作 - 「月次レポート」= sales_monthly.csv → monthly_report.docx - 「競合分析」 = competitor_info.txt + product_spec.txt → positioning_report.docx - 「レビュー分析」= customer_reviews.csv → review_report.docx ## 禁止事項 - 既存ファイルを上書きしない（新規ファイル名に日付を付ける） - ダミーデータで補完しない（データが不足なら確認する）
書くべき内容と書かなくていい内容
✗ 書かなくていい
✓ 書くべき
「丁寧に答えてください」 「日本語で話してください」 「あなたはアナリストです」 → セッション毎に指示すればいい → CLAUDE.md は永続するので 人格・口調より「プロジェクト固有の事実」を書く
ファイル名と各ファイルの内容 出力先フォルダのルール 数値フォーマット（万円・小数点桁数） 文字コード・フォントの指定 プロジェクト固有の用語定義
ポイント
「毎回説明していること」がCLAUDE.mdに書く候補。セッションをまたぐたびに「このフォルダには…」と説明し直しているなら、それは CLAUDE.md の仕事。逆に毎回変わる条件（「今日は6月データだけ見たい」）は都度指示する。この区別ができると生産性が大きく変わる。
Topic 02 ワークフロー定義
ワークフロー定義とは：「どのファイルを読み→どう処理し→何を出力する」という一連の手順をテキストファイルに書いたもの。Claude Code に渡すと、書いた通りに自律実行してくれる。
なぜ使うか：毎月同じ手順を繰り返す作業（月次レポート・競合モニタリング）をファイルに書いておけば、来月は「このワークフローを実行して」の一言で完了する。
Mission 2-A ／ Workflow
月次レポート生成ワークフローをテキストファイルで定義する
シナリオ
月次レポートを毎月手作業で作っているが、やることは毎回同じ。①売上CSV読み込み → ②チャネル別集計 → ③グラフ生成 → ④Wordレポート出力 の4ステップを workflows/monthly_report.md に書いて、来月からはそのファイルを渡すだけにしたい。
作成するファイル
d2c-analytics/
├── CLAUDE.md
├── workflows/
│   └── monthly_report.md   ← 今回作る
└── ...
タスク ／ 成果物
  • ワークフロー定義ファイル workflows/monthly_report.md を作成するmonthly_report.md
  • ステップは①〜④を明示し、各ステップで「入力ファイル・処理内容・出力ファイル」を書く
  • エラー時の挙動（処理を止めて確認する）もステップとして含める
着目ポイント：ワークフロー定義は「人間が読んでも理解できる」粒度で書く。Claude Code が読むだけでなく、チームメンバーが見ても何をしているかわかる仕様書になるのが理想。
模範 workflows/monthly_report.md
monthly_report.md# ワークフロー：月次売上レポート生成 ## 目的 sales_monthly.csv から経営陣向けの月次レポートを自動生成する。 ## 入力ファイル - sales_monthly.csv（d2c-analytics/ 直下） ## 出力ファイル - outputs/monthly_report_YYYYMM.docx（日付を自動付与） - outputs/graphs/channel_trend_YYYYMM.png - outputs/graphs/category_trend_YYYYMM.png ## 実行ステップ ### Step 1: データ読み込みと検証 - sales_monthly.csv を読み込む - 列名が正しいか確認（月, チャネル, 商品カテゴリ, 売上金額, 注文件数, 新規顧客数, リピート顧客数） - 異常値（売上が0以下、NaN）があれば処理を止めて確認を求める ### Step 2: 集計処理 - チャネル別・月別の売上金額を集計 - 前月比を計算（最新月 vs 前月） - 直近6ヶ月の移動平均を計算 ### Step 3: グラフ生成 - チャネル別月次売上推移（折れ線）→ channel_trend_YYYYMM.png - カテゴリ別月次売上（積み上げ棒）→ category_trend_YYYYMM.png - 日本語フォント対応、軸ラベルは日本語、万円単位 ### Step 4: Wordレポート生成 - エグゼクティブサマリー（3行）を自動生成 - グラフを貼り付け - チャネル別評価コメントを追記 - outputs/monthly_report_YYYYMM.docx として保存 ## エラー処理 - Step 1 でデータ異常を検出した場合：処理を中断し、異常箇所を報告して指示を待つ - Step 3 でフォント未インストールの場合：英語フォントで続行して警告を出す
ポイント
ワークフロー定義の3要素は「入力・処理・出力」。各ステップでこの3つが明確なら Claude Code は迷わず動ける。特に出力ファイル名に YYYYMM のような動的部分を含めると、毎月実行しても上書きされない安全な設計になる。
Mission 2-B ／ Workflow
ワークフロー定義ファイルを Claude Code に渡して実行させる
シナリオ
2-A で作った monthly_report.md を、実際に Claude Code に渡して実行させる。ここでのゴールは「ワークフロー定義ファイルを読んで、全ステップを自律実行してもらう」こと。途中で止まったときの対処も体験する。
タスク ／ 成果物
  • Claude Code に workflows/monthly_report.md を読ませて全ステップを実行させる月次レポート一式
  • 実行中に Claude Code が「確認を求めてきた場合」の対処を体験する
  • 出力ファイルが outputs/ に正しく保存されたか確認する
着目ポイント：ワークフロー定義が曖昧だと Claude Code が途中で止まる。どのステップで止まったかをメモしておき、定義を改善するフィードバックループを回すこと。
Claude Code への模範指示文
ターミナル入力workflows/monthly_report.md を読み込んで、 書かれているステップをすべて実行してください。 実行前に以下を確認してください： - sales_monthly.csv が存在するか - outputs/ フォルダが存在しなければ作成する 各ステップが完了したら「Step X 完了」と報告してから次に進んでください。 エラーが発生した場合はすぐに停止して原因を教えてください。
止まったときの対処パターン
対話例# Claude が止まって確認を求めてきた場合 Claude: 「Step 1 で売上金額が 0 の行が 2件あります。 処理を続けますか？除外しますか？」 → 対処A：続行する場合 「0の行は除外して処理を続けてください」 → 対処B：確認したい場合 「どの行かを表示してから判断します」 # ワークフロー定義が曖昧で止まった場合 Claude: 「グラフのサイズが指定されていません。 デフォルト（1200x800px）で進めますか？」 → ワークフロー定義に追記する： 「グラフサイズ：1200x800px、dpi:150」
ポイント
「止まった箇所＝定義に書くべきこと」。Claude Code が確認を求めてくるたびにワークフロー定義を更新していくと、数回で「一切止まらず最後まで動く」完成版ができあがる。これがワークフロー定義の育て方。
Topic 03 Skills（スキル定義）
Skill とは：Claudeに特定のタスクやワークフローの処理方法を教える「命令セット」。フォルダとしてパッケージ化され、中に SKILL.md（YAMLフロントマター付きMarkdown）を必ず含む。CLAUDE.md がプロジェクト全体の設定なら、Skill は「特定タスクの専門家」。
作り方の流れ：① まず「何をSkill化するか」ユースケースを2〜3個洗い出す → ② スキル名/SKILL.md のフォルダを作る（フォルダ名はケバブケース） → ③ YAMLフロントマター（name + description）を書く → ④ 本文に処理手順を書く → ⑤ テストして改善する。
段階的開示：フロントマターの description は常にClaudeに読み込まれる（第1レベル）。SKILL.md 本文は「関連する」と判断されたときだけ読み込まれる（第2レベル）。さらに references/ フォルダに詳細ドキュメントを置ける（第3レベル）。これにより多数のスキルを登録してもトークン消費を最小限に抑えられる。
便利ツール：skill-creator スキル（Claude.ai に組み込み済み / Claude Code 用にも入手可能）を使えば、自然言語の説明からスキルのドラフトを自動生成し、レビュー・改善まで支援してもらえる。
Skill の作法・ルール一覧
命名規則
  • ファイル名は必ず SKILL.md（大文字小文字厳密。skill.md SKILL.MD は不可）
  • フォルダ名はケバブケースのみ：review-analysis ✅ ／ review_analysis ❌ ／ ReviewAnalysis ❌
  • name に claude や anthropic は使えない（予約語）
YAMLフロントマターの制限
  • description は 1024文字以内
  • XMLタグ（< >）は禁止（フロントマターはシステムプロンプトに入るため）
  • YAML内でのコード実行は不可
ファイル構成
  • スキルフォルダ内に README.md を置かない（ドキュメントは SKILL.md か references/ へ）
  • SKILL.md 本文は 5,000語以内に収める（超える分は references/ フォルダへ移動）
  • 任意フォルダ：scripts/（実行コード）、references/（参考ドキュメント）、assets/（テンプレートなど）
配布する場合
  • GitHubで配布する際はリポジトリレベルの README は必要（スキルフォルダ内のREADMEとは別）
  • インストール手順・サンプル使用例を README に記載する
  • オープンソースにする場合はフロントマターに license: MIT などを追加
  • アップロード時は zip 形式 に圧縮する
Mission 3-A ／ Skills
レビュー分析 Skill を作成して、ワークフローから呼び出す
シナリオ
Skill を作る前に「何をSkill化すべきか」を考える。今回のプロジェクトでは ①レビュー分析 ②競合分析 ③売上分析 が繰り返し使う処理＝Skill 候補。まず①レビュー分析から、以下の推奨テンプレートに沿って skills/review-analysis/SKILL.md を手書きで作ってみる。
推奨構造テンプレート（この型に沿って書く）
---
name: your-skill-name
description: 機能の説明。ユーザーが[具体的なフレーズ]を
  尋ねるときに使用します。
---

  • description には「何をするか」＋「ユーザーが言いそうなフレーズ」を両方含める

着目ポイント：テンプレートの各セクションには意味がある。description は自動トリガーの判断基準、手順 は処理の再現性、出力フォーマット は品質の一貫性、トラブルシューティング はエラー時の自己解決。全部揃って初めて「信頼できるスキル」になる。
模範 skills/review-analysis/SKILL.md

[テーマ]（根拠：R001, R005） 2. [テーマ]（根拠：R003） 3. [テーマ]（根拠：R007, R010） ■ ネガティブ TOP3 1. [テーマ]（根拠：R006, R012） ... ■ 属性別サマリー | 軸 | ポジ | ネガ | 主なコメント | |---------|------|------|------------| | 価格 | X件 | X件 | ... | ... ``` ## 例 ### 例1：全件分析 ユーザーが言う: 「レビューを分析して」 結果: 全レビューを対象に分析サマリーを生成 ### 例2：期間指定 ユーザーが言う: 「直近3ヶ月のレビュー傾向を教えて」 結果: 対象期間を絞って分析サマリーを生成 ## トラブルシューティング ### CSV が読み込めない 原因: ファイルパスの変更、またはエンコーディングの問題 解決策: CLAUDE.md でファイルパスを確認。UTF-8 以外の場合はエンコーディングを指定する ### 根拠レビューIDが見つからない 原因: CSV にID列がない、または列名が異なる 解決策: CSV の列名を確認し、手順の「レビューID」を実際の列名に合わせる ## 出力先 ワークフローから呼ばれた場合 → ワークフローの出力先に従う 単独実行の場合 → outputs/review_analysis_YYYYMMDD.md
ワークフローからの呼び出し方
monthly_report.md に追記する例### Step 2.5: レビュー分析の挿入 review-analysis スキルを使って 顧客レビューデータを分析し、 結果を月次レポートの「顧客フィードバック」セクションに挿入する。
ポイント
Skill は「部品」、Workflow は「設計図」、CLAUDE.md は「プロジェクト仕様書」。この3層構造が揃うと、Claude Code は設計図を読んで部品を組み合わせながら自律的に仕事を進められる。1つのSkillを複数のWorkflowから再利用することで、定義の重複がなくなり保守が楽になる。

YAMLフロントマターの段階的開示：フロントマター（name + description）は常にClaudeのシステムプロンプトに読み込まれるが、SKILL.md本文は「関連する」と判断されたときだけ読み込まれる。これにより、多数のスキルを登録してもトークン消費を最小限に抑えられる。
Mission 3-B ／ Skills（応用）
競合モニタリング Skill を作り、パラメータを渡して呼び出す
シナリオ
競合データの分析を Skill 化する。今回は Mission 3-A で学んだ SKILL.md の構造に加え、「分析する競合ブランド」や「比較軸」をパラメータとして渡せる設計にする。
ポイント：YAMLフロントマターの description にパラメータの説明も含めると、Claude が「このスキルにはどんな引数を渡せるか」を自動で理解する。
作成するファイル
d2c-analytics/
├── skills/
│   ├── review-analysis/
│   │   └── SKILL.md
│   └── competitor-analysis/    ← ケバブケース
│       └── SKILL.md            ← 今回作る
└── ...
タスク ／ 成果物
  • skills/competitor-analysis/SKILL.md を作成し、YAMLフロントマター（name: competitor-analysis）を書くSKILL.md
  • description に「競合分析」「ブランド比較」などのトリガーフレーズと、受け取れるパラメータ（brands, axes）を記述する
  • SKILL.md 本文にパラメータの受け取り方・処理手順・出力フォーマットを記述する
  • 「Aブランドと Bブランドだけ価格と成分で比較して」という呼び出しで動作確認する競合比較レポート
着目ポイント：パラメータを受け取る Skill を作ると「汎用ツール」になる。パラメータなしの Skill は「決まった処理をするマクロ」。どちらが向いているかはユースケースによって変わる。description にパラメータの存在を書いておくと、Claudeがユーザーの指示から自動的に引数を抽出して渡してくれる。
模範 skills/competitor-analysis/SKILL.md
SKILL.md--- name: competitor-analysis description: 競合ブランドと自社商品を比較分析します。brandsパラメータで分析対象を、 axesパラメータで比較軸を指定できます。ユーザーが「競合を比較して」 「ブランド比較」「競合分析して」と言うときに使用します。 --- # 競合分析 ## パラメータ - brands：分析対象ブランド名（複数可、未指定＝全ブランド） - axes：比較軸（価格 / 成分濃度 / チャネル / ターゲット） ## 入力ファイル - 競合情報データ（必須） - 自社商品仕様（自社比較に使用） ## 処理 1. 指定された brands のデータを競合情報から抽出 2. 指定された axes の情報を各ブランドから取り出す 3. 自社（ビタグロウ）のデータを商品仕様から取り出す 4. 指定 axes を列とした比較表を作成する 5. 自社の相対的なポジションにコメントを付ける ## 出力フォーマット - 比較表（Markdown テーブル） - 自社ポジションの総評（3行以内） - 推奨アクション（1〜2点） ## 出力先 outputs/competitor_analysis_YYYYMMDD.md
ポイント
パラメータは「使うたびに変わる条件」に使う。「どのブランドを比較するか」は毎回変わる可能性があるのでパラメータ向き。「出力先フォルダ」は毎回変わらないので CLAUDE.md に書くべき。

フロントマターの description にパラメータを含める理由：Claudeはフロントマターだけを先に読み込む（段階的開示の第1レベル）。description に「brandsパラメータで対象を指定」と書いておけば、ユーザーが「AとBを比較して」と言ったとき、Claudeが自動でこのスキルを選び、SKILL.md本文を読み込んで実行する。
Mission 3-C ／ Skills（実践）
skill-creator で売上分析 Skill を生成し、テスト・改善する
シナリオ
3-A・3-B では手動で SKILL.md を書いた。実務では skill-creator スキルを使うと、自然言語で「こんなスキルが欲しい」と伝えるだけでドラフトを自動生成できる。今回は売上分析スキルを skill-creator で生成し、トリガーテストと改善まで一気に行う。
タスク ／ 成果物
  • Claude に「skill-creator スキルを使って、月次売上データの分析スキルを作って」と依頼し、ドラフトを生成させるSKILL.md ドラフト
  • 生成された SKILL.md の description を確認し、ユーザーが言いそうなフレーズ（「売上を分析して」「月次データをまとめて」など）が含まれているかチェックする
  • トリガーテスト：「売上の傾向を教えて」「先月の売上レポート作って」など言い換えた表現で呼び出し、スキルが自動で読み込まれるか確認する
  • トリガーされなかった場合は description を改善して再テストする改善済み SKILL.md
着目ポイント：skill-creator は「15〜30分でスキルを作れる」ツールだが、重要なのはその後のテストと改善。特に description の書き方でスキルの使い勝手が大きく変わる。「曖昧すぎる description」と「具体的なトリガーフレーズを含む description」の差を体感してほしい。
skill-creator への依頼プロンプト
Claude への指示skill-creator スキルを使って、以下のスキルを作成してください。 ## ユースケース 月次売上データ（チャネル別・カテゴリ別）を分析し、 トレンド・前月比・チャネル構成比をサマリーにまとめる。 ## トリガーされるべきフレーズ - 「売上を分析して」「月次データをまとめて」 - 「売上レポート作って」「チャネル別の傾向を教えて」 ## 出力フォーマット - 全体サマリー（前月比・トレンド） - チャネル別構成比（表形式） - 注目ポイント3つ
description の良い例 vs 悪い例
✗ 悪い例（曖昧・トリガーなし）
✓ 良い例（具体的・トリガーフレーズあり）
--- name: sales-analysis description: 売上データを分析します。 --- → 曖昧すぎてClaudeが いつ使うか判断できない
--- name: sales-analysis description: 月次売上データを分析し、 チャネル別・カテゴリ別のトレンドと 前月比をサマリーにまとめます。 ユーザーが「売上を分析して」 「月次データをまとめて」 「チャネル別の傾向を教えて」 と言うときに使用します。 --- → 何をするか＋いつ使うかが明確
トリガーテストの方法
テスト手順## トリガーすべきフレーズ（成功を確認） ✅ 「売上を分析して」 ✅ 「先月の売上データをまとめて」 ✅ 「チャネル別の売上傾向を教えて」 ## トリガーすべきでないフレーズ（誤作動しないことを確認） ❌ 「レビューを分析して」→ review-analysis が起動すべき ❌ 「競合を比較して」→ competitor-analysis が起動すべき ❌ 「Instagram投稿文を作って」→ スキル不要 ## 失敗した場合の改善方法 description にトリガーフレーズを追加するか、 より具体的な表現に変える。 過剰トリガーする場合は 「シンプルなデータ閲覧には使用しない」 などネガティブトリガーを追加する。
ポイント
Skill づくりの最短ルートは「skill-creator で生成 → テスト → description 改善」の反復。手書きでゼロから作るより、skill-creator にドラフトを任せて自分はテストと改善に集中するのが効率的。特に description は「ユーザーが実際に言いそうな言い回し」を入れることが最も重要。ここが不十分だとスキルが眠ったままになる。
Topic 04 スラッシュコマンド
スラッシュコマンドとは：/report のように / で始まるショートカット。長い指示文をコマンド一発に圧縮できる。.claude/commands/ フォルダに Markdown ファイルを置くだけで定義できる。
典型的な使い方：/monthly と打つだけで月次レポート生成ワークフローが走る。/review でレビュー分析が走る。繰り返し使う操作をコマンドにまとめることで入力ミスもなくなる。
Mission 4-A ／ Slash Commands
/report と /review スラッシュコマンドを定義して呼び出す
シナリオ
毎月のルーティンを2つのコマンドに圧縮する。/report で月次レポートワークフロー全体が走り、/review でレビュー分析 Skill が実行される。コマンドファイルは .claude/commands/ に置く。
作成するファイル
d2c-analytics/
├── CLAUDE.md
├── .claude/
│   └── commands/
│       ├── report.md    ← /report コマンド
│       └── review.md    ← /review コマンド
├── workflows/
└── skills/
タスク ／ 成果物
  • .claude/commands/report.md を作成して /report コマンドを定義するreport.md
  • .claude/commands/review.md を作成して /review コマンドを定義するreview.md
  • Claude Code で /report と入力して、月次レポートが自動実行されることを確認する動作確認
着目ポイント：スラッシュコマンドは「ワークフローやSkillへの入口」として機能する。コマンドファイルの中身はシンプルでいい。「どのワークフロー/Skillを呼ぶか」と「引数をどう渡すか」だけ書けば十分。
模範 .claude/commands/report.md
report.md（/report コマンド）# /report 月次売上レポートを生成します。 workflows/monthly_report.md を読み込んで、 書かれているすべてのステップを実行してください。 引数が渡された場合（例：/report 2024-06）： - 指定された年月のデータのみを対象にする - 出力ファイル名に指定年月を使う 引数がない場合： - 最新月（CSVの最後の月）を対象にする
模範 .claude/commands/review.md
review.md（/review コマンド）# /review カスタマーレビューを分析します。 review-analysis スキルを実行してください。 引数が渡された場合（例：/review 30代）： - 指定された年代のレビューのみを対象にする 引数がない場合： - 全レビューを対象にする
ポイント
コマンド・Skill・Workflowの3層を正しく分離する。コマンドは「起動ボタン」、Skill は「処理の部品」、Workflow は「処理の設計図」。コマンドに処理の中身を全部書くのは避ける。呼び出す先（Workflow or Skill）を指定するだけにするとメンテナンスが楽になる。
Topic 05 Hooks（自動トリガー）
Hook とは：「特定のイベントが起きたとき自動的に実行される処理」のこと。ファイル保存時・コマンド実行前後・エラー発生時など、イベントに紐づけて Claude Code の動作をカスタマイズできる。
典型的な使い方：出力ファイルが保存されるたびに自動でバリデーション（数値が正常か確認）を走らせる。デプロイ前に自動テストを実行させる。これにより「確認忘れ」を構造的に防げる。
Mission 5-A ／ Hooks
outputs/ への保存時に自動バリデーションする Hook を設定する
シナリオ
月次レポートを自動生成すると、たまに売上合計が前月より異常に大きい/小さいデータが混入することがある。outputs/ に CSV が保存されるたびに自動で数値チェックを走らせ、異常値があれば警告を出す Hook を設定する。
作成するファイル
d2c-analytics/
├── CLAUDE.md
├── .claude/
│   ├── commands/
│   └── hooks/
│       └── validate_output.md    ← 今回作る
└── ...
タスク ／ 成果物
  • .claude/hooks/validate_output.md に Hook 定義を作成するvalidate_output.md
  • トリガー：outputs/*.csv が生成・更新されたとき
  • チェック内容：売上金額が 0 以下の行がないか・前月比が 300% 超の異常値がないか
  • CLAUDE.md に Hook の存在を記載して、セッションをまたいでも有効になるようにする
着目ポイント：Hook は「後工程の品質保証」として使う。自動生成で起きやすいミス（異常値・文字化け・フォーマット崩れ）を人間が見る前に捕まえる仕組みを作ること。
模範 .claude/hooks/validate_output.md
validate_output.md# Hook: 出力ファイルのバリデーション ## トリガー outputs/*.csv が新規作成または更新されたとき ## 実行する検証 ### 検証1: 数値の妥当性チェック - 売上金額列に 0以下の値がないか確認する - 前月比が 300%を超える値がないか確認する - NaN / null / 空文字がないか確認する ### 検証2: フォーマットチェック - ヘッダー行が正しいか（期待するカラム名と一致するか） - 日付列のフォーマットが "YYYY年M月" 形式か ## 結果の報告 問題なし： 「✓ validate_output: [ファイル名] の検証が完了しました。異常なし。」 問題あり： 「⚠ validate_output: [ファイル名] で以下の問題を検出しました。 - [問題の内容と行番号] 処理を続けますか？（続行 / 修正してから再実行）」 ## CLAUDE.md への追記 CLAUDE.md の「出力ルール」に以下を追加してください： 「outputs/に CSV を保存するたびに .claude/hooks/validate_output.md の Hook が自動実行されます」
ポイント
Hook は「自動検査員」として設計する。人間が毎回目視確認するのではなく、起きやすいミスのパターンを Hook に書いておく。一度書けば以後ずっと自動でチェックされる。「どんなミスが起きやすいか」のリストが Hook の設計書になる。
Topic 06 Subagents（サブエージェント）
Subagent とは：メインの Claude Code が「別の Claude エージェント」を起動して並列処理させる仕組み。「レビュー分析」と「競合分析」を同時進行させることで、逐次処理より大幅に時間を短縮できる。
なぜ使うか：互いに依存しない処理は並列化できる。1エージェントが1つのタスクに集中するため、コンテキストの混濁（データが混ざるバグ）も防げる。
Mission 6-A ／ Subagents
競合分析・レビュー分析・売上分析を Subagent で並列実行するワークフローを設計する
シナリオ
Q3施策立案のために競合分析・レビュー分析・売上分析の3つを同時に行いたい。これらは互いに独立しているので、3つの Subagent に同時並列で実行させ、全部終わったら結果を統合して施策ロードマップを作る。
タスク ／ 成果物
  • 並列実行のワークフロー workflows/q3_strategy.md を設計するq3_strategy.md
  • 3つの処理を Subagent として定義し、それぞれに担当 Skill を割り当てる
  • 全 Subagent 完了後に結果を統合して施策ロードマップを生成するステップを含める
着目ポイント：「並列化できる処理」と「逐次にしなければならない処理」を見極めること。統合ステップは全 Subagent の完了を待ってから実行する（依存関係の管理）。
模範 workflows/q3_strategy.md
q3_strategy.md# ワークフロー：Q3 施策立案（Subagent 並列版） ## 目的 3種類の分析を並列実行し、Q3 施策ロードマップを生成する。 ## Phase 1: 並列分析（Subagent 3本を同時起動） 以下の3つのSubagentを同時に起動してください： ### Subagent A — 売上分析 担当スキル: sales-analysis（作成済みのものを使用） 入力: 月次売上データ 出力: outputs/subagent_sales_YYYYMMDD.md ### Subagent B — レビュー分析 担当スキル: review-analysis 入力: 顧客レビューデータ 出力: outputs/subagent_review_YYYYMMDD.md ### Subagent C — 競合分析 担当スキル: competitor-analysis 入力: 競合情報データ、自社商品仕様 出力: outputs/subagent_competitor_YYYYMMDD.md ## Phase 2: 統合（全Subagent完了後に実行） Subagent A・B・C がすべて完了したことを確認してから以下を実行する： 1. 3つの出力ファイルを読み込む 2. 各分析から「最重要インサイト2点」を抽出する 3. 以下の3軸で施策を整理する： - チャネル改善（売上分析起点） - 商品改良（レビュー分析起点） - 競合対策（競合分析起点） 4. 各施策に優先度・担当チーム・KPI・実施月を付ける 5. outputs/q3_roadmap_YYYYMMDD.xlsx として保存する
並列化できる処理 / できない処理
✗ 逐次にすべき（依存関係あり）
✓ 並列化できる（独立）
「レビュー分析の結果を受けて 競合分析の切り口を変える」 ← 前の処理の結果を 次の処理が使うケース
「レビュー分析」 「競合分析」 「売上分析」 ← 3つが互いに無関係で 同じ最終出力（施策）に向かうケース
ポイント
Subagent の設計は「何を独立させて、何を統合するか」の判断が全て。Phase 1 は並列、Phase 2 は全完了待ち逐次、という2フェーズ構造が Subagent を使う典型パターン。統合ステップでコンテキストが混ざらないよう、各 Subagent の出力を別ファイルに書かせるのが重要。
Topic 07 Tasks ／ MCP
Tasks とは：Claude Code をバックグラウンドで定期実行・スケジュール実行させる仕組み。「毎月1日に月次レポートを自動生成」のような定期処理を設定できる。
MCP (Model Context Protocol) とは：Claude Code が外部サービス（Slack・Notion・Google Drive など）と接続するための標準プロトコル。MCP サーバーを接続すると、Claude Code が「Slack にメッセージを送る」「Notion にデータを書き込む」などの外部連携が可能になる。
Mission 7-A ／ Tasks
月次レポート生成を毎月1日に自動実行する Task を設定する
シナリオ
毎月1日に手動で /report を実行していたのを自動化したい。CLAUDE.md に Task 定義を追加して、毎月1日の朝9時に月次レポートが自動生成されるように設定する。
タスク ／ 成果物
  • CLAUDE.md に Task セクションを追加して定期実行を定義するCLAUDE.md（更新版）
  • Task の内容：毎月1日 09:00 に workflows/monthly_report.md を実行
  • 実行完了後に outputs/ の最新ファイル一覧をコンソールに出力するステップを追加する
着目ポイント：Task は「誰もトリガーしなくても動く」設計にする。エラーが発生しても次のアクション（通知 or 再試行）を定義しておかないと、気づかないまま失敗し続ける。
CLAUDE.md への Task セクション追記例
CLAUDE.md（追記部分）## Tasks（定期実行） ### monthly-report スケジュール：毎月1日 09:00 実行内容：workflows/monthly_report.md を全ステップ実行する 完了後：outputs/ 内の最新ファイル一覧をターミナルに出力する エラー発生時： - エラーログを outputs/logs/error_YYYYMMDD.log に保存する - 処理を中断して原因をログに記録する - 次回の手動実行時にログを読んで問題を報告する
ポイント
Task はエラー処理とセットで設計する。自動実行は「誰も見ていないときに動く」ので、失敗したときの記録・通知をあらかじめ決めておく必要がある。エラーログをファイルに書き出す設計にすると、次に手動実行したときに Claude Code が「前回こういうエラーがありました」と報告してくれる。
Mission 7-B ／ MCP
MCP で Slack と接続し、レポート完了時に自動通知するワークフローを作る

  • workflows/monthly_report.md に Slack 通知ステップを追加するmonthly_report.md（更新版）

CLAUDE.md への MCP セクション追記例
CLAUDE.md（追記部分）## MCP 接続サービス ### Slack 接続名：slack-mcp 用途：レポート完了通知・エラーアラート 通知先チャンネル：#marketing-reports 通知に含める情報： - 生成ファイル名 - 生成日時 - エグゼクティブサマリー（1行）
monthly_report.md への Slack 通知ステップ追記例
monthly_report.md（追記ステップ）### Step 5: Slack 通知（MCP） slack-mcp を使って #marketing-reports チャンネルに 以下のフォーマットで通知を送る： 📊 月次レポートが生成されました ファイル：[出力ファイル名] 生成日時：[YYYY-MM-DD HH:MM] サマリー：[Step 4 で生成したエグゼクティブサマリー1行] 送信後「Step 5 完了: Slack 通知送信済み」と報告する。

全体設計のまとめ
7トピックで作り上げた構造の全体像。

CLAUDE.md ── プロジェクトの記憶。常に読まれる設定書。
Workflow ── 処理の設計図。ステップ定義と依存関係を管理。
Skills ── YAMLフロントマター付きSKILL.mdで定義する再利用可能な処理部品。段階的開示で自動トリガーされる。
Slash Commands ── 頻繁に使う操作の起動ボタン。
Hooks ── 自動品質保証。ファイル保存時などに自動で走る検査。
Subagents ── 並列処理の設計。独立した分析を同時に走らせる。
Tasks + MCP ── 自律化の完成形。スケジュール実行＋外部連携。

この7層が揃ったとき、Claude Code は「対話型ツール」から「自律動作するワークフローエンジン」に変わる。
